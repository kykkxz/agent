## 模块一：http协议

### 1.生活化的例子：

http协议：浏览器和服务器之间的“密码”，

HyperText Transfer Protocol 超文本传输协议：规定浏览器和服务器是怎么进行对话的一个规则

### 2.http协议的工作流程

分四步

```
1. 建立连接（打电话） 
2. 发送请求（说话） 
3. 接收响应（听对方回答） 
4. 关闭连接（挂电话）
```

### 3. http请求的组成

- 请求行：告诉服务器，用什么方法进行请求，用什么语言，找谁

- 请求头：告诉服务器，我是谁，我想要什么格式，其他需求

- 请求体：给到服务器的具体朱提（post等才有）

##### 3.1请求行

方法/路径/协议版本

比如：GET/index.html HTTP/1.1

- 方法：告诉服务器要干什么

- 路径：我要找哪个资源

- 协议版本：用什么语言

##### 常用的 HTTP 方法

| 方法      | 大白话解释      | 生活例子              | 是否幂等* | 爬虫常用度 |
| ------- | ---------- | ----------------- | ----- | ----- |
| GET     | "给我看看"     | 去书店翻书看            | ✅ 是   | ⭐⭐⭐⭐⭐ |
| POST    | "帮我存一下"    | 把信投进邮筒            | ❌ 否   | ⭐⭐⭐⭐  |
| PUT     | "帮我更新/替换"  | 修改已提交的表格          | ✅ 是   | ⭐     |
| DELETE  | "帮我删掉"     | 撕掉已提交的表格          | ✅ 是   | ⭐     |
| PATCH   | "帮我修改一部分"  | 在表格上只改一个错别字       | ❌ 否   | ⭐     |
| HEAD    | "给我看看文件信息" | 问书店"这本书有没有货"      | ✅ 是   | ⭐⭐    |
| OPTIONS | "你能干什么？"   | 问服务员"你们店支持什么支付方式" | ✅ 是   | ⭐     |

> *幂等：同一个请求发一次和发一百次，效果一样。比如 GET 看网页，看一百次也不会改变服务器数据。但 POST 提交表单，发一百次就创建了100条记录。



**GET 和 POST 的完整区别对比**（重点！面试常考）：

| 对比维度       | GET                       | POST                       |
| ---------- | ------------------------- | -------------------------- |
| **用途**     | 获取数据（只读）                  | 提交数据（可修改）                  |
| **数据传输位置** | 参数拼在 URL 后面（`?key=value`） | 参数放在请求体（Body）里             |
| **数据长度限制** | 有！URL 最长约 2048 字符         | 理论上没有限制（服务器可能有配置上限）        |
| **安全性**    | 低！参数暴露在 URL 里，会被浏览器历史记录保存 | 高！参数在请求体里，不会出现在 URL 中      |
| **缓存**     | 可以被浏览器缓存                  | 默认不缓存                      |
| **书签**     | 可以收藏为书签（因为 URL 包含所有信息）    | 不能收藏为书签                    |
| **编码类型**   | 只能是 ASCII 字符（中文要 URL 编码）  | 支持多种编码（application/json 等） |
| **幂等性**    | ✅ 幂等（多次请求效果相同）            | ❌ 不幂等（多次请求可能创建多条记录）        |
| **典型场景**   | 搜索、翻页、查看文章详情              | 登录、注册、上传文件、提交表单            |

## 4.HTTP响应的组成：

- 状态行：是否成功，用的什么协议

- 响应头：数据格式，大小，缓存规则等

- 响应体：HTML网页，JSON数据，图片，视频等

### 4.1 状态行

格式： 协议版本 状态码 状态描述

比如：`HTTP/1.1 200 OK`

- 协议版本：服务器用的http协议版本（HTTP/1.1 或HTTP/2）

- 状态码：三位数，告诉你请求的结果

- 状态描述：状态码的文字说明（仅做参考）

#### 4.2状态码

**状态码分类速记表**：

| 范围  | 分类名   | 一句话总结         | 生活比喻             |
| --- | ----- | ------------- | ---------------- |
| 1xx | 信息性响应 | "我知道了，正在处理"   | 服务员说"好的，您稍等"     |
| 2xx | 成功    | "你要的东西给你了"    | 菜端上来了            |
| 3xx | 重定向   | "东西不在这儿，去别处找" | "这个菜没了，你去隔壁窗口看看" |
| 4xx | 客户端错误 | "你的请求有问题"     | "你点的菜我们菜单上没有"    |
| 5xx | 服务器错误 | "我的锅，我自己出问题了" | "厨房着火了，厨师罢工了"    |

##### 1xx 信息性响应（很少见）

这类状态码表示请求已接收，继续处理。一般不会在爬虫中遇到，因为浏览器/库会自动处理。

| 状态码 | 含义                  | 说明                         |
| --- | ------------------- | -------------------------- |
| 100 | Continue            | "继续发吧，我准备好了"               |
| 101 | Switching Protocols | "我们要切换协议了"（如升级到 WebSocket） |

##### 2xx 成功（皆大欢喜）

| 状态码 | 含义         | 大白话             | 爬虫场景           |
| --- | ---------- | --------------- | -------------- |
| 200 | OK         | "你要的东西给你了"      | 最常见，说明请求成功     |
| 201 | Created    | "你让我创建的东西已经建好了" | POST 创建资源成功后返回 |
| 204 | No Content | "操作成功了，但没啥返回给你" | DELETE 删除成功后常见 |

##### 3xx 重定向（换个地方找）

| 状态码 | 含义                | 大白话               | 区别            |
| --- | ----------------- | ----------------- | ------------- |
| 301 | Moved Permanently | "我永久搬家了，新地址是 XXX" | 浏览器会自动缓存这个新地址 |
| 302 | Found             | "我临时搬了，新地址是 XXX"  | 下次可能还回老地址     |
| 304 | Not Modified      | "你要的内容没变，用你缓存的吧"  | 浏览器缓存机制，节省流量  |

> **爬虫注意**：requests 默认自动跟随重定向。如果你想禁止自动跳转，可以设置 `allow_redirects=False`。

##### 4xx 客户端错误（你的问题）

| 状态码 | 含义                 | 大白话          | 常见原因              |
| --- | ------------------ | ------------ | ----------------- |
| 400 | Bad Request        | "你说的我听不懂"    | 请求格式错误、参数缺失       |
| 401 | Unauthorized       | "请先登录"       | 没有携带认证信息          |
| 403 | Forbidden          | "你不配看这个"     | 有身份但权限不够，或触发反爬机制  |
| 404 | Not Found          | "你要的东西不存在"   | URL 写错了，或页面已删除    |
| 405 | Method Not Allowed | "这个方法不支持"    | 比如该用 POST 你用了 GET |
| 408 | Request Timeout    | "你说话太慢了"     | 请求发送超时            |
| 429 | Too Many Requests  | "你问得太快了，歇会儿" | 触发频率限制（反爬常见）      |

> **爬虫最怕的两个 4xx**：
> 
> * **403**：通常意味着你被识别为爬虫了。解决方案：换 User-Agent、加 Cookie、降低请求频率、用代理 IP。
> * **429**：请求太频繁被封了。解决方案：加 `time.sleep()` 延时、用代理 IP 池。

##### 5xx 服务器错误（服务器的问题）

| 状态码 | 含义                    | 大白话        | 常见原因           |
| --- | --------------------- | ---------- | -------------- |
| 500 | Internal Server Error | "厨房着火了"    | 服务器代码有 Bug     |
| 502 | Bad Gateway           | "中间人传话传错了" | 代理服务器或负载均衡器出问题 |
| 503 | Service Unavailable   | "暂时不营业"    | 服务器维护或超载       |
| 504 | Gateway Timeout       | "中间人等太久了"  | 上游服务器响应超时      |

> **爬虫策略**：遇到 5xx 不要急着放弃，等一会儿重试可能就好了。可以用 `time.sleep()` 等几秒再试。

**状态码速记口诀**：

* 2 开头：成功！开心！
* 3 开头：换地方！
* 4 开头：你错了！
* 5 开头：他错了！

### 4.3 响应头：服务器给你的附加信息

键:值

**常见的响应头字段：**

①Content-Type：最重要的响应头之一，返回的是什么格式的数据

```
# HTML 网页
Content-Type: text/html; charset=utf-8

# JSON 数据
Content-Type: application/json; charset=utf-8

# 图片
Content-Type: image/png

# 文件下载
Content-Type: application/octet-stream
```

② Content-Length：返回多少字节

```
Content-Length: 12345
```

③Set-Cookie：服务器允许客户端保存的Cookie，发送下次请求的时候客户端会自动带上

```
Set-Cookie: session=abc123; Path=/; HttpOnly; Secure
```

- Path=/ ：这个Cookie对整个网站都有效

- HttpOnly：JavaScript无法读取

- Secure： 只能通过https传输

④ Cache-Control ：缓存控制，告诉客户端这个内容可以缓存多久

```
# 不缓存
Cache-Control: no-store

# 缓存 1 小时
Cache-Control: max-age=3600

# 可以缓存，但使用前要验证
Cache-Control: no-cache
```

⑤Server：告诉客户端用的是什么服务器软件

```
Server: nginx/1.24.0
Server: Apache/2.4.58
Server: gunicorn/21.2.0
```

**注意**：隐藏本字段，暴露服务器版本会成为攻击弥补表

⑥Content-Encoding：用什么方法压缩

```
Content-Encoding: gzip
```

⑦Location：重定向地址，返回状态码3xx，告诉客户端“新地址”在哪里

```
HTTP/1.1 301 Moved Permanently
Location: https://www.new-domain.com/page
```

```
HTTP/1.1 200 OK
```

## 3. requests库完整参数

requests 的每个方法（`get()`、`post()`、`put()`、`delete()` 等）都支持一系列参数。掌握这些参数，你就能应对绝大多数爬虫场景。

**requests 支持的方法一览**：

| 方法                   | 用途         | 类比            |
| -------------------- | ---------- | ------------- |
| `requests.get()`     | 获取数据       | 去书店看书         |
| `requests.post()`    | 提交数据       | 寄信/填表         |
| `requests.put()`     | 完整更新资源     | 把整张表重新填一遍     |
| `requests.patch()`   | 部分更新资源     | 只改表上的一个错别字    |
| `requests.delete()`  | 删除资源       | 撕掉已提交的表格      |
| `requests.head()`    | 只获取响应头     | 问"这本书有没有货"    |
| `requests.options()` | 询问服务器支持的方法 | 问"你们支持什么支付方式" |

**每个方法的通用参数**（以 `get()` 和 `post()` 为例）：

```python
requests.get(url, 
    params=None,       # URL 查询参数
    headers=None,      # 请求头
    cookies=None,      # Cookie
    auth=None,         # 认证信息
    timeout=None,      # 超时设置
    proxies=None,      # 代理
    data=None,         # 请求体（POST 等）
    json=None,         # JSON 请求体（POST 等）
    allow_redirects=True,  # 是否允许重定向
    verify=True,       # 是否验证 SSL 证书
    stream=False       # 是否流式下载
)
```

①url：目标地址，必须是“”，字符串，必须加http://或者https://

②params：url查询参数，专门给get方法用，把字典自动拼接

```python
# 等价于 https://www.baidu.com/s?wd=python&rn=10
response = requests.get("https://www.baidu.com/s", 
    params={"wd": "python", "rn": 10})
```

**POST/PUT/PATCH请求专用data 和 json 的区别**（面试常考！）：

| 对比维度             | `data` 参数                           | `json` 参数                  |
| ---------------- | ----------------------------------- | -------------------------- |
| **Content-Type** | `application/x-www-form-urlencoded` | `application/json`         |
| **数据格式**         | 表单格式：`key=value&key2=value2`        | JSON 格式：`{"key": "value"}` |
| **适用场景**         | 传统网页表单提交                            | 现代 API 调用                  |
| **编码**           | 自动 URL 编码                           | 自动 `json.dumps()` 序列化      |

网页登录、表单提交：data

接口调用、前后端API交互：json

```python
# demo_对比_手动 Cookie 与 Session.py
# 演示 requests 中“手动传递 Cookie"与"Session 对象管理 Cookie"的区别
# 目标：通过 httpbin.org 测试接口，直观展示两种方式的差异

import requests

def demo_manual_cookie():
    """
    演示 1：手动传递 Cookie
    适用场景：只需临时发送一次 Cookie，不需要后续请求复用
    """
    print("="*50)
    print(" 演示一：手动传递 Cookie (一次性)")
    print("="*50)

    url = "http://httpbin.org/get"
    
    # 准备我们要发送的 Cookie (字典格式)
    # 就像你去游乐园，手里拿了一张单次体验券
    my_cookies = {
        "user_id": "10086",
        "login_status": "guest"
    }
    
    print(f"📡 正在发送请求到: {url}")
    print(f" 携带的 Cookie: {my_cookies}")
    
    # 使用 requests.get 的 cookies 参数发送
    response = requests.get(url, cookies=my_cookies)
    
    if response.status_code == 200:
        data = response.json()
        # httpbin.org/get 会在 headers 字段中返回它收到的 Cookie
        received_cookie = data['headers'].get('Cookie', '无')
        print(f"✅ 请求成功！")
        print(f" 服务器看到的 Cookie: {received_cookie}")
    else:
        print("❌ 请求失败")


def demo_session_cookie():
    """
    演示 2：使用 Session 对象管理 Cookie
    适用场景：需要维持登录状态，跨请求共享 Cookie (如爬虫登录后的后续操作)
    """
    print("\n" + "="*50)
    print("🛒 演示二：使用 Session 对象 (持久化)")
    print("="*50)

    # 1. 创建一个 Session 对象
    # 这相当于你办理了一张游乐园 VIP 年卡，以后每次来都能自动识别你的身份
    session = requests.Session()
    
    # 2. 设置 Cookie (或者通过第一次请求由服务器下发 Cookie)
    # 这里我们模拟：先向一个“设置 Cookie"的接口发送请求
    url_set_cookie = "http://httpbin.org/cookies/set/session_id/998877"
    print(f" 步骤 A: 首次请求设置 Cookie -> {url_set_cookie}")
    session.get(url_set_cookie)
    
    # 3. 发起后续请求 (Session 会自动带上之前设置的 Cookie)
    url_get_info = "http://httpbin.org/get"
    print(f"📡 步骤 B: 发起后续请求 -> {url_get_info}")
    response_1 = session.get(url_get_info)
    
    if response_1.status_code == 200:
        data = response_1.json()
        received_cookie = data['headers'].get('Cookie', '无')
        print(f"✅ 第一次请求成功！服务器识别到 Cookie: {received_cookie}")

    # 4. 再次发起请求 (证明 Cookie 被 Session 记住了)
    print(f"📡 步骤 C: 再次发起请求 (无需重新设置 Cookie) -> {url_get_info}")
    response_2 = session.get(url_get_info)
    
    if response_2.status_code == 200:
        data = response_2.json()
        received_cookie = data['headers'].get('Cookie', '无')
        print(f"✅ 第二次请求成功！服务器依然识别到 Cookie: {received_cookie}")

if __name__ == "__main__":
    demo_manual_cookie()
    demo_session_cookie()
    print("\n🎉 演示结束。对比输出结果，你会发现 Session 方式下的 Cookie 是持久存在的。")
```

#### 3.Xpath解析网页

Xpath：xml path language 

[XPath 简介 | 菜鸟教程](https://www.runoob.com/xpath/xpath-intro.html)

常用 XPath 语法：

| XPath 表达式             | 含义                |
| --------------------- | ----------------- |
| `//div`               | 所有 div 元素         |
| `//div[@class="job"]` | class 为 job 的 div |
| `//h2/text()`         | h2 标签里的文字         |
| `//a/@href`           | a 标签的 href 属性值    |
| `//ul/li[1]`          | ul 下第一个 li        |

Python 用 `lxml` 库解析 XPath（需要安装：`pip install lxml`）：

```python
# 解析html文档
from lxml import etree
html = """
<html>
  <body>
    <div class="job">
      <h2>Python工程师</h2>
      <p class="salary">20-30K</p>
      <p class="city">北京</p>
    </div>
    <div class="job">
      <h2>Java开发</h2>
      <p class="salary">15-25K</p>
      <p class="city">上海</p>
    </div>
  </body>
</html>
"""

#解析HTML
tree = etree.HTML(html)

# 提取所有职位
jobs = tree.xpath("//div[@class='job']")
for job in jobs:
    print("========", job.xpath(".//h2/text()"))
    title = job.xpath(".//h2/text()")[0]
    salary = job.xpath(".//p[@class='salary']/text()")[0]
    city = job.xpath(".//p[@class='city']/text()")[0]
    print(f"标题: {title}, 薪资: {salary}, 城市: {city}")
```
